# AIH Test Project Goals

Created: 2026-07-12

## Purpose

Define practical AIH tests for evaluating agentic AI hallucination,
state-fidelity, evidence handling, timing, uncertainty, and task suitability.

## Goals

1. Define bounded AIH tests that can be run and scored.

2. Keep each test narrow enough to preserve evidence and compare agents under
   the same conditions.

3. Start with AIchess as the lowest-level formal state test.

4. Develop an intermediate certificate-style AIH test after the basic harness
   pattern is proven.

5. Treat k-phd as the broad knowledge-ladder extension, not the first
   implementation target.

6. Record raw responses, parsed responses, timing, errors, human assistance,
   and statistics where repeated runs support them.

7. Use public repositories as curated project archives containing reviewed
   current state and rebuildable test definitions.

8. Treat optimization claims as test variables, not conclusions. RAG,
   retrieval tuning, semantic cache, KV/v-cache behavior, memory cache,
   reranking, vector-store choice, and other LinkedIn/research-sourced
   recommendations should be tested only through bounded AIH runs.

   The local AI/ML optimization research subproject inside HybridAI should
   supply candidate tuning methods, but AIH should decide whether they help by
   running bounded tests with external scoring.

9. Track hardware environment as a future project variable. The immediate AIH
   goal is to learn what runs in the current Brilliance test environment, but
   later runs should record and compare higher-core, higher-RAM, GPU,
   external-GPU, Thunderbolt/advanced-I/O, storage, and power/thermal profiles
   when hardware upgrades become available.

## Current Test Ladder

```text
AIchess -> intermediate AIH certificate-style test -> k-phd
```

## Hallucination Mode Taxonomy

The AIH project should use the shared hallucination-mode taxonomy in:

```text
docs/AIH_HALLUCINATION_MODE_TAXONOMY_20260713.md
```

The taxonomy treats hallucination as a workflow disturbance where an agent or
human acts from an unsupported, false, stale, or evidence-resistant model of
the current state.

This means human oversight is part of the reliability problem, not a perfect
external safety layer. A hallucination mode that applies to an AI agent can
also apply to human review, project summaries, and human-agent workflow pairs.

## Optimization Variables To Test

AIH should be able to compare the same test with different stack options:

- default unmodified repository/software behavior before any tuning;
- no retrieval vs provided-source-packet retrieval;
- basic RAG vs tuned RAG;
- vector search vs hybrid sparse/dense retrieval;
- reranking on/off;
- semantic cache on/off;
- KV/v-cache and memory-cache options on/off where supported;
- software/runtime switches changed one at a time on fixed hardware;
- Ollama vs vLLM vs Hugging Face serving paths;
- different model versions, including Qwen variants;
- Codex, Claude/Anthropic, and other cloud-agent runs as comparison baselines;
- future hardware environment changes, once available, recorded as explicit
  test variables rather than hidden improvements;
- different agentic-AI top layers using the same test prompt and scoring
  contract.

The goal is not to assume any tuning method is better. The goal is to observe
whether a tuning option improves parseability, correctness, timing, source
discipline, and reproducibility under a fixed AIH test.

Every optimization test should begin with a default baseline run in the current
Brilliance test environment before any repository-software modification, tuning
flag, cache layer, retrieval layer, or serving-layer change is introduced.

Codex should be tracked as a moving comparison baseline. Claude/Anthropic and
other advanced cloud agentic-AI products should be added where available. Each
cloud-agent run should record the date, product surface, model identity where
available, tool access, prompt envelope, and result so later runs can show
whether the cloud stack changed over time.

## Non-Goals

- Do not claim that one AI stack is generally better from one bounded result.
- Do not publish raw logs, failed attempts, caches, raw audio, or scratch runs.
- Do not expand the test suite faster than tests can actually be executed.
