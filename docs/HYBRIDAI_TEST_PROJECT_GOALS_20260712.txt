# HybridAI Test Project Goals

Created: 2026-07-12

## Purpose

Use AIH tests to define, implement, and evaluate practical hallucination tests
under bounded, externally scored conditions.

The primary value is the test methodology: formal task design, deterministic
referees, evidence preservation, invalid-output classification, and measured
state-fidelity failure.

Testing many stack configurations is secondary. Broad stack benchmarking can be
left to the larger industry unless a configuration comparison directly improves
or validates the hallucination test itself.

## Goals

1. Treat HybridAI as a hallucination-test development environment first, and as
   one possible system under test second.

2. Make each HybridAI version able to accept a bounded test, produce parseable
   output, and preserve enough evidence for external scoring.

3. Compare HybridAI against itself, a selected reference agent, and peer
   candidate AI stacks only where the comparison clarifies the hallucination
   test or validates the test contract.

4. Start with modest AIchess runs such as one-move probes and short smoke games
   before repeated statistical runs.

5. Record timing, model identity, prompt, tool access, reference access,
   parsing behavior, scoring, errors, and statistics.

6. Use results to guide the next HybridAI implementation.

7. Publish only the curated rebuildable current state of each subproject
   version: source, stable scripts, selected configuration, dependency notes,
   test definitions, reviewed result summaries, and relevant planning links.

8. Keep stack-configuration dimensions explicit so later comparisons are
   possible, but do not make broad stack sweeps the main project objective.
   Build HybridAI so specific agentic-AI stack implementations can vary by
   top layer, middle layer, retrieval/cache/memory layer, serving layer, and
   model version while still running the same AIH test contract.

9. Plan for future hardware environment upgrades as a separate comparison
   dimension. More CPU cores, more RAM, local GPU, external GPU modules,
   Thunderbolt/advanced-I/O paths, and faster storage may become necessary for
   larger RAG, cache, vLLM, or Hugging Face runs, but they should be recorded as
   environment changes rather than mixed into software-tuning claims.

10. Create a local AI/ML optimization research subproject inside HybridAI. This
    subproject should collect, summarize, and test local inference, retrieval,
    cache, memory, serving, and classical-ML optimization options before they
    are promoted into production HybridAI stack implementations.

11. Add realistic professional time deadlines and projected project progress to
    active work plans. For each time box, record the target deadline, expected
    deliverables, likely evidence artifacts, and explicit non-promises. This
    keeps the project closer to professional research/engineering practice than
    open-ended exploratory prompting.

## Minimum Useful HybridAI Run

```text
test definition -> agent response -> external referee/grader -> structured result
```

## Time-Box Planning Rule

HybridAI work should use explicit time-boxes when moving from exploration into
execution.

Each time-box should state:

- deadline;
- intended scope;
- expected deliverables;
- expected evidence artifacts;
- stopping conditions;
- what is explicitly out of scope for that deadline.

The point is not to force false certainty. The point is to make projected
progress auditable and to distinguish realistic professional delivery from
unbounded AI-assisted exploration.

## Stack Dimensions

HybridAI stack implementations should be described by explicit dimensions:

1. Top layer: LangGraph, CrewAI, AutoGen, OpenHands, LlamaIndex/Agno, or a
   HybridAI-native implementation.

2. Middle layer: direct Ollama API, Ollama OpenAI-compatible `/v1`, LiteLLM,
   TinyProxy, custom AIH transport shim, and logging/telemetry path.

3. Retrieval/cache/memory layer: source-packet only, RAG, tuned RAG,
   reranking, vector store, semantic cache, KV/v-cache behavior, memory cache,
   and other cache or retrieval options identified from research or LinkedIn
   recommendations.

4. Bottom/supporting layer: Ollama, vLLM, Hugging Face/Transformers/TGI, and
   later llama.cpp or other local serving paths.

5. Model: model family, version, quantization, context length, parameters,
   and runtime notes.

6. Hardware environment: current laptop/workstation profile first, with later
   upgraded systems recorded by CPU cores, RAM, GPU/external GPU, high-speed
   I/O path, storage, operating system, and power/thermal limits.

## Optimization Candidates

HybridAI should eventually test options such as:

- default unmodified repository/software behavior before tuning;
- RAG vs no-RAG;
- chunking strategy;
- embedding model;
- vector store;
- sparse/dense hybrid retrieval;
- reranking;
- semantic cache;
- KV/v-cache and memory-cache options;
- vLLM serving/cache behavior;
- Hugging Face serving/cache behavior;
- Ollama model/version differences;
- Codex, Claude/Anthropic, and other cloud-agent comparison runs;
- future hardware-upgrade runs where larger local stacks require more CPU,
  RAM, GPU, external GPU, high-speed I/O, or storage bandwidth;
- classical ML baselines such as XGBoost or LightGBM where the task is
  structured prediction or grading rather than natural-language generation.

## Local AI/ML Optimization Research Subproject

Purpose:

Learn which local AI/ML optimization methods are relevant to HybridAI before
treating them as implementation defaults.

Research inputs may include:

- official project documentation;
- academic papers;
- benchmark reports;
- LinkedIn posts and technical discussions;
- local experiment results;
- comparison runs against Codex, Claude/Anthropic, and other cloud products.

Initial LinkedIn research-intake example:

- XGBoost / LightGBM technical discussion:
  https://www.linkedin.com/posts/lanchuhuong_xgboost-and-lightgbm-remains-my-go-to-ml-activity-7475169821475823618-pv-V/

Candidate topics include:

- RAG design;
- chunking and source-packet construction;
- embedding models;
- vector stores;
- sparse/dense hybrid retrieval;
- reranking;
- semantic cache;
- KV/v-cache behavior;
- memory cache;
- Ollama tuning;
- vLLM serving behavior;
- Hugging Face/Transformers/TGI serving behavior;
- quantization;
- context-length tradeoffs;
- software/runtime switches on fixed hardware;
- structured-output constraints;
- XGBoost/LightGBM and other classical ML baselines for scoring, grading, or
  structured-prediction tasks.

Research rule:

An optimization recommendation should become a test candidate first, not a
default implementation. Each candidate should be compared against a default
baseline under at least one bounded AIH test before it is treated as an
improvement.

Controlled-switch rule:

When possible, test software/runtime switches while holding hardware fixed.
Change one setting at a time, record the default setting, the modified setting,
the model/version, the serving path, and the AIH result. Examples may include
context length, quantization, batch/concurrency settings, cache settings,
retrieval settings, reranker settings, decoding parameters, timeout limits, and
tool-access settings.

The default baseline matters. Before modifying repository software, enabling
cache/RAG layers, changing serving paths, or tuning model parameters, HybridAI
should record a default run for the same AIH test. Later tuned runs should be
compared against that baseline rather than against memory or impression.

HybridAI results should also be compared against Codex, Claude/Anthropic, and
other cloud-agent runs where practical. Because cloud products can evolve over
time, each comparison should record the run date, observed model/product
surface, tool access, prompt envelope, and AIH result rather than assuming
continuity across dates.

## Non-Goals

- Do not use public repositories as full local work histories.
- Do not publish build debris, scratch logs, raw transcripts, or failed
  preliminary attempts by default.
- Do not claim broad superiority over Codex, cloud agents, or local stacks
  without comparable bounded results.
