# HybridAI Test Project Goals

Created: 2026-07-12

## Purpose

Use AIH tests to evaluate HybridAI versions as agent implementations under
bounded, externally scored conditions.

## Goals

1. Treat HybridAI as a system under test, not as the judge of its own output.

2. Make each HybridAI version able to accept a bounded test, produce parseable
   output, and preserve enough evidence for external scoring.

3. Compare HybridAI against itself, a selected reference agent, and peer
   candidate AI stacks where practical.

4. Start with modest AIchess runs such as one-move probes and short smoke games
   before repeated statistical runs.

5. Record timing, model identity, prompt, tool access, reference access,
   parsing behavior, scoring, errors, and statistics.

6. Use results to guide the next HybridAI implementation.

7. Publish only the curated rebuildable current state of each subproject
   version: source, stable scripts, selected configuration, dependency notes,
   test definitions, reviewed result summaries, and relevant planning links.

8. Build HybridAI so specific agentic-AI stack implementations can vary by
   top layer, middle layer, retrieval/cache/memory layer, serving layer, and
   model version while still running the same AIH test contract.

9. Plan for future hardware environment upgrades as a separate comparison
   dimension. More CPU cores, more RAM, local GPU, external GPU modules,
   Thunderbolt/advanced-I/O paths, and faster storage may become necessary for
   larger RAG, cache, vLLM, or Hugging Face runs, but they should be recorded as
   environment changes rather than mixed into software-tuning claims.

## Minimum Useful HybridAI Run

```text
test definition -> agent response -> external referee/grader -> structured result
```

## Stack Dimensions

HybridAI stack implementations should be described by explicit dimensions:

1. Top layer: LangGraph, CrewAI, AutoGen, OpenHands, LlamaIndex/Agno, or a
   HybridAI-native implementation.

2. Middle layer: direct Ollama API, Ollama OpenAI-compatible `/v1`, LiteLLM,
   TinyProxy, custom AIH transport shim, and logging/telemetry path.

3. Retrieval/cache/memory layer: source-packet only, RAG, tuned RAG,
   reranking, vector store, semantic cache, KV/v-cache behavior, memory cache,
   and other cache or retrieval options identified from research or LinkedIn
   recommendations.

4. Bottom/supporting layer: Ollama, vLLM, Hugging Face/Transformers/TGI, and
   later llama.cpp or other local serving paths.

5. Model: model family, version, quantization, context length, parameters,
   and runtime notes.

6. Hardware environment: current laptop/workstation profile first, with later
   upgraded systems recorded by CPU cores, RAM, GPU/external GPU, high-speed
   I/O path, storage, operating system, and power/thermal limits.

## Optimization Candidates

HybridAI should eventually test options such as:

- default unmodified repository/software behavior before tuning;
- RAG vs no-RAG;
- chunking strategy;
- embedding model;
- vector store;
- sparse/dense hybrid retrieval;
- reranking;
- semantic cache;
- KV/v-cache and memory-cache options;
- vLLM serving/cache behavior;
- Hugging Face serving/cache behavior;
- Ollama model/version differences;
- Codex, Claude/Anthropic, and other cloud-agent comparison runs;
- future hardware-upgrade runs where larger local stacks require more CPU,
  RAM, GPU, external GPU, high-speed I/O, or storage bandwidth;
- classical ML baselines such as XGBoost or LightGBM where the task is
  structured prediction or grading rather than natural-language generation.

The default baseline matters. Before modifying repository software, enabling
cache/RAG layers, changing serving paths, or tuning model parameters, HybridAI
should record a default run for the same AIH test. Later tuned runs should be
compared against that baseline rather than against memory or impression.

HybridAI results should also be compared against Codex, Claude/Anthropic, and
other cloud-agent runs where practical. Because cloud products can evolve over
time, each comparison should record the run date, observed model/product
surface, tool access, prompt envelope, and AIH result rather than assuming
continuity across dates.

## Non-Goals

- Do not use public repositories as full local work histories.
- Do not publish build debris, scratch logs, raw transcripts, or failed
  preliminary attempts by default.
- Do not claim broad superiority over Codex, cloud agents, or local stacks
  without comparable bounded results.
