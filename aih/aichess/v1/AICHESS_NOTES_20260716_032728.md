## Flags

```text
-nb   number of boards
-nl   number of board-loop batches
-gmto game timeout seconds
-mt   move timeout seconds
-mxply maximum plies
-cnrtlm cumulative correction/rejection limit
-ans  agent-only; no Stockfish validation when no Stockfish validation flag is set
-avb  Stockfish validates board/rules; referee agent validates moves
-avm  Stockfish validates moves; referee agent validates board
```

`-ans`, `-avb`, and `-avm` short single-dash forms are accepted as aliases for
`--ans`, `--avb`, and `--avm`.

For `-ans` positional agents:

```text
1 agent: agent fills white, black, referee
2 agents: both are players, second also fills referee
3 agents: first two are players, third fills referee
```

## Output format

Move field is fixed 5 characters.

```text
e2e4  -a   accepted
b1c1  -il  parsed UCI move rejected as illegal
----- -iv  invalid output; parser found no UCI move
TIMEO -f   timeout/forfeit
```

Per-line counters are fixed 3 digits:

```text
il=000 iv=000 nr=000
```

End-of-board counter line:

```text
nil=000 niv=000 ncr=050
```

Meanings:

```text
il/nil = illegal parsed UCI moves
iv/niv = invalid/unparseable player outputs
nr     = il + iv
ncr    = configured cumulative rejection limit
```

## Important findings

The model can describe UCI legality in broad terms, but under move pressure it
confuses strict UCI syntax with chess-like short notation.

Adding this reinforcement to every player move request helped format validity:

```text
Return a UCI move of at least 4 characters.
```

This fixed the repeated short-output loop in some runs:

```text
e4
e3
```

After that, `-ans` shifted from invalid-output failures to UCI-shaped illegal
move failures.


This means:

## Stockfish-backed behavior

With Stockfish validation:

```bash
./aichess.sh -nb 1 -gmto 300 -cnrtlm 50 qwen1
```

The agent reached legal moves, then failed around a repeated illegal move:

```text
b1c1
```

Important parser issue observed:

```text
raw="b1c1 is illegal in this position, so I'll suggest a different..."
```

The parser currently extracts the first UCI-shaped token, so it can re-extract
the rejected move from explanatory correction text. Next fix to consider:

```text
When parsing correction responses, ignore the previously rejected move and parse
the next UCI-shaped token, or tighten correction prompts to require only the
replacement move.
```

## Determinism finding

With `temperature=0`, the same prompt and same board state produce repeatable
failures. This is useful: a hallucination can become a deterministic probe.

Changing the prompt changes the deterministic output path. Example:

```text
normal prompt -> illegal/invalid move
correction prompt -> sometimes recovers
```

## Correction note

Always prompt the player for a UCI move of at least 4 characters.

Always search returned text for UCI-shaped tokens, but when handling a correction
response, do not accept the already-invalidated UCI move as the replacement
move. If the returned string includes the invalidated UCI move, keep scanning
for another UCI-shaped token before deciding that the correction failed.
